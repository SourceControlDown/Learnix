## ADR-LMS-001: Course як єдиний Aggregate Root для структури

**Рішення:** Сутність `Course` є Aggregate Root (коренем агрегату) для `Section` та `Lesson`. Всі операції зі зміни структури курсу (додавання секцій, уроків, їх видалення чи зміна порядку) виконуються виключно через публічні методи класу `Course` (наприклад, `Course.AddSection()`, `Course.RemoveLesson()`). Мутаційні методи всередині `Section` та `Lesson` позначені як `internal` і доступні лише в межах збірки Domain.

**Чому:**
- **Захист інваріантів публікації (Publish Invariants):** Опублікований курс має суворі правила (повинен мати обкладинку, хоча б одну секцію, хоча б один видимий урок). Якщо дозволити редагувати секції/уроки напряму через репозиторій, можна випадково видалити останній урок опублікованого курсу і зламати систему. Коли все йде через `Course`, агрегат перевіряє ці правила при кожній дії.
- **Єдине місце збереження:** Зміни в структурі автоматично оновлюють `UpdatedAt` курсу.
- **Спрощення обробників команд:** Application layer працює за єдиним патерном: дістав курс → викликав метод → зберіг. Базовий клас `CourseCommandHandler` інкапсулює цю логіку.

**Альтернативи що розглядались:**
- **Окремі репозиторії для Section та Lesson:** Дозволяють швидше точково оновлювати записи, але повністю руйнують консистентність курсу. Відкинуто згідно з принципами Domain-Driven Design (DDD).

## ADR-LMS-002: Table Per Hierarchy (TPH) для типів уроків

**Рішення:** Різні типи уроків (`VideoLesson`, `PostLesson`, `TestLesson`) успадковуються від абстрактного базового класу `Lesson` і зберігаються в єдиній таблиці бази даних `Lessons` з використанням паттерну TPH (Table Per Hierarchy). EF Core автоматично використовує колонку-дискримінатор `LessonType`.

**Чому:**
- **Ефективність читання:** При завантаженні структури курсу (всіх секцій та уроків) EF Core робить лише один `JOIN` до таблиці `Lessons`, незалежно від того, скільки типів уроків існує.
- **Спільні дані:** Усі уроки мають багато спільного (`Title`, `Order`, `SectionId`, `IsFreePreview`, `CreatedAt`), що ідеально лягає в одну таблицю. Специфічні поля (наприклад, `VideoBlobPath` або `TestConfiguration`) просто залишаються `NULL` для інших типів.

**Альтернативи що розглядались:**
- **Table Per Type (TPT):** Зберігання базових даних у `Lessons`, а специфічних — у таблицях `VideoLessons`, `TestLessons`. Відкинуто через проблему `N+1 JOIN` при читанні повного списку уроків курсу, що критично для продуктивності API.
- **Окремі сутності без успадкування:** Руйнує загальний порядок сортування (Order) всередині секції. Відкинуто.

## ADR-LMS-003: Value Objects як JSONB для системи тестування

**Рішення:** Структура тестів (`Question`, `QuestionOption`, `TextAnswerConfig`) та відповіді студентів (`StudentAnswer`) спроєктовані як **Value Objects** і зберігаються в базі даних PostgreSQL у форматі **JSONB** (колонка в таблиці `Lessons` для конфігурації, і колонка в таблиці `TestAttempts` для відповідей). Вони не мають власних сурогатних Primary Keys (ID) і окремих SQL-таблиць.

**Чому:**
- **Документо-орієнтована природа тестів:** Питання і відповіді ідеально лягають у структуру JSON. У тестах немає потреби робити SQL `JOIN` або пошук по окремому варіанту відповіді. Тест завжди завантажується цілком і валідується цілком.
- **Уникнення Table Bloat:** Якби питання та опції були окремими таблицями, це створило б 4 нові таблиці та величезну кількість непотрібних зв'язків. JSONB залишає схему БД чистою.
- **Інкапсуляція логіки:** Логіка перевірки тесту інкапсульована в домені: `TestLesson.CalculateScore(List<StudentAnswer>)`. 

**Альтернативи що розглядались:**
- **Повна реляційна модель (Questions, Options, Answers):** Правильно з точки зору класичних БД, але надмірно складно для платформи, де тести є лише частиною контенту (read-heavy сценарій). Відкинуто.
- **Зберігання в MongoDB:** Оскільки MongoDB вже налаштована (для чатів та відгуків), можна було б зберігати тести там. Проте, тести прив'язані до транзакцій створення курсу в PostgreSQL. Зберігання структури тесту в Mongo порушило б цілісність даних (ACID) курсу.

## ADR-LMS-004: Ідемпотентне відстеження прогресу (LessonProgress)

**Рішення:** Прогрес проходження фіксується на рівні кожного уроку створення запису в таблиці `LessonProgress`. Статус проходження всього курсу (`Enrollment.Status = Completed`) не є окремою дією користувача, а обчислюється і встановлюється автоматично (через Domain Events) у момент, коли користувач завершує останній обов'язковий урок курсу.

**Чому:**
- **Атомарність:** Студент завершує урок → `LessonProgress` зберігається → генерується подія `LessonCompletedDomainEvent` → система перевіряє, чи залишились ще незавершені уроки в цій реєстрації (`Enrollment`).
- **Ідемпотентність:** Повторний запит на завершення уроку просто ігнорується (запис `LessonProgress` вже існує), що убезпечує від мережевих ретраїв.
- **Гнучкість при оновленнях курсу:** Якщо інструктор додає новий урок в уже опублікований курс, відсоток проходження існуючих студентів (які пройшли, наприклад, 10/10) автоматично адаптується (стане 10/11), і система зрозуміє це без складних міграцій статусу `Enrollment`.

**Альтернативи що розглядались:**
- **Зберігання масиву пройдених уроків у JSONB в Enrollment:** Спрощує базу, але ускладнює агрегаційні запити (наприклад, "скільки студентів пройшли цей конкретний урок"). Окрема таблиця `LessonProgress` ефективніша для аналітики.
