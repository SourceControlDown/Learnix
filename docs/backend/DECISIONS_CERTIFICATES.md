## ADR-CERT-001: Бібліотека QuestPDF для генерації PDF

**Рішення:** Для створення макетів та рендерингу PDF-сертифікатів використовується бібліотека **QuestPDF**. Макет описується повністю через C# Fluent API (code-first підхід), без використання проміжних HTML-шаблонів.

**Чому:**
- **Швидкодія та споживання пам'яті:** QuestPDF працює надзвичайно швидко (генерація простого сертифіката займає < 50 мс) і має дуже низьке споживання оперативної пам'яті, оскільки це native .NET рішення, яке використовує SkiaSharp під капотом.
- **Відсутність зовнішніх залежностей:** На відміну від рішень на базі wkhtmltopdf або Puppeteer, QuestPDF не вимагає встановлення браузерів (Chrome/Edge) або важких бінарників на хост-машині/в Docker-контейнері.
- **Fluent API:** Макет жорстко типізований. Компілятор перевіряє код макета, що робить підтримку та рефакторинг набагато безпечнішими порівняно з HTML/CSS стрінгами.
- **Точність верстки:** Дозволяє працювати з абсолютними координатами, точними розмірами шрифтів та міліметрами, що критично важливо для сертифікатів, які можуть друкуватися на папері формату А4.

**Альтернативи що розглядались:**
- **HTML-to-PDF (DinkToPdf / wkhtmltopdf):** Застаріла технологія, часто ламає верстку, вимагає наявності native C++ бібліотек у системі.
- **Puppeteer-Sharp (headless Chrome):** Чудово рендерить HTML, але тягне за собою сотні мегабайт Chromium. Для on-demand API це занадто повільно і "важко" (startup time браузера).
- **iText7 / PDFsharp:** Потужні, але мають складніший (іноді застарілий) API та суворі ліцензійні обмеження (AGPL для iText). QuestPDF (на умовах Community License) ідеально підходить для цього проекту.

## ADR-CERT-002: Генерація QR-кодів через QRCoder

**Рішення:** Для генерації QR-коду на сертифікаті (який веде на публічну сторінку валідації сертифіката) використовується бібліотека **QRCoder**.

**Чому:**
- **Інтеграція з QuestPDF:** QuestPDF не має вбудованого генератора QR-кодів. Він приймає готові зображення (масиви байтів). QRCoder — це надійна, lightweight C#-бібліотека, яка легко генерує матрицю QR-коду і перетворює її на PNG/Byte Array.
- **Офлайн генерація:** Дозволяє створювати QR-коди повністю in-memory без звернення до зовнішніх API, що гарантує стабільність, швидкість і приватність.
- **Відсутність прив'язки до System.Drawing:** QRCoder підтримує генерацію базових графічних форматів без використання `System.Drawing.Common`, який є проблемним на Linux/Docker (починаючи з .NET 6).

**Альтернативи що розглядались:**
- **Зовнішні сервіси (напр., Google Chart API / goqr.me):** Потребують інтернет-з'єднання, створюють затримку в мережі, можуть впасти або змінити ліміти. Заборонено для синхронної генерації сертифікатів.
- **ZXing.Net:** Дуже потужна бібліотека для читання/запису баркодів. Трохи перевантажена (overkill) функціоналом, якщо потрібен лише простий QR-код. QRCoder є значно легшим.

## ADR-CERT-003: On-Demand (синхронна) генерація сертифікатів

*(Також відображено в DECISIONS_INFRA.md)*

**Рішення:** PDF сертифікат генерується синхронно безпосередньо під час HTTP-запиту користувача (`POST /api/certificates/courses/{courseId}/generate`). Фоновий воркер для генерації не використовується.

**Чому:**
- **Швидкість QuestPDF:** Оскільки генерація займає мілісекунди, немає сенсу виносити це в background queue.
- **UX:** Користувач натискає кнопку "Отримати сертифікат" і миттєво отримує файл. Фонова генерація змушувала б його дивитись на лоадер "Generating..." і чекати.
- **Спрощення архітектури:** Відпадає необхідність керувати станами сертифіката (`Pending`, `Generated`, `Failed`) та зберігати тимчасові файли на диску перед завантаженням в Azure Blob Storage.

**Альтернативи що розглядались:**
- **MassTransit Consumer / Background Service:** Розглядалось спочатку, але відкинуто як overkill після того, як тестування показало, що QuestPDF справляється із задачею "на льоту" без впливу на thread pool API сервера.
