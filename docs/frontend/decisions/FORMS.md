# Learnix — Frontend Architecture Decision Records (Forms)

> Format: Decision → Why → Alternatives.

---

## ADR-FRONT-FORMS-001: React Hook Form & Lightweight Wrappers

**Decision:**
- We use `react-hook-form` (RHF) as the core library for managing form state and validation.
- Instead of using heavy UI wrappers or Context providers (like the ones generated by `shadcn/ui`), we use our own lightweight, custom components (e.g., `FormInput`, `FormTextarea`, `PasswordInput` in `src/components/common/form/`).
- These components use `forwardRef` to integrate directly with RHF's `register()` function.

**Why:**
- **Performance:** RHF uses uncontrolled components, minimizing re-renders. Bypassing heavy React Context wrappers keeps the component tree shallow and fast.
- **Simplicity:** Our custom wrappers (`FormInput`) are under 40 lines of code, making them extremely easy to debug, extend, and style with pure Tailwind CSS.
- **Bundle Size:** Zero dependency on `shadcn/ui` Radix primitives means less JavaScript shipped to the client.

**Alternatives:**
- `shadcn/ui` Form wrappers: Discarded because the overhead of `FormProvider` and deeply nested components was deemed unnecessary for our standard forms.
- Formik: Discarded due to performance issues with large forms (it relies heavily on controlled inputs and re-renders the whole form on every keystroke).

## ADR-FRONT-FORMS-002: Zod Schemas as Source of Truth

**Decision:**
- Zod schemas define the structure and validation rules for all forms.
- The TypeScript `FormValues` type is inferred directly from the Zod schema (`z.infer<typeof schema>`).
- DTOs (Data Transfer Objects) mapping to the backend are defined as separate standard TypeScript interfaces in `src/types/`.
- The transformation from `FormValues` to `DTO` occurs explicitly in the form's `onSubmit` handler.

**Why:**
- The shape of a form often differs from the API contract (e.g., a comma-separated string of tags in the UI vs. an array of strings in the DTO).
- Trying to force a single type for both UI and API leads to awkward compromises. Explicit mapping in `onSubmit` is highly visible and easy to debug.

**Consequences:**
- Zod is strictly for form validation via `@hookform/resolvers/zod`. We do *not* use Zod to validate incoming API responses to avoid runtime overhead.
- `src/types/` contains manually written interfaces for DTOs, grouped by domain.

## ADR-FRONT-FORMS-003: Server-to-Client Validation Mapping

**Decision:**
- We handle server-side validation failures (HTTP 400 Bad Request) at the Field-Level.
- The `isValidationError` guard and `setApiFieldErrors` utility (`src/utils/errors.ts`) parse the backend's `ProblemDetails` (RFC 7807) `errors` dictionary.
- These errors are mapped directly onto the `react-hook-form` fields using the `setError` function. Unmapped errors fall back to a `root` form error.

**Why:**
- A server validation error needs to show the user exactly which input failed, directly under the input field, rather than hidden in a generic global popup.
- `ProblemDetails` is the ASP.NET Core standard, making the mapping predictable.

## ADR-FRONT-FORMS-004: Form Errors vs Global Errors

**Decision:**
- By default, unhandled API errors trigger a global `sonner` Toast via the `QueryClient` `onError` handler (Application/Business Level).
- For form mutations, we explicitly use the `suppressGlobalError` escape hatch (`meta: { suppressGlobalError: true }`).

**Why:**
- Without suppression, a backend validation failure would trigger BOTH a red text message under the specific input field AND a large red Toast notification in the corner of the screen.
- Suppressing the global toast for form mutations keeps the UX clean and focused entirely on the form itself.
